# eFPS Intelligence — Backend System Design & PRD
**Version:** 1.0 | **Scale Target:** 1,000,000 users | **Type:** Government FPS (Fair Price Shop) Management Platform

---

## 1. What This System Actually Is

eFPS Intelligence is a **Fair Price Shop (Ration Shop) Management Portal** used by shop owners (dealers) operating under Gujarat's PDS (Public Distribution System). It solves operational problems these dealers face daily:

- Logging into the government's FPS portal is slow, session-heavy, and crash-prone
- Dealers need real-time transaction visibility (who collected ration, who didn't)
- Stock reconciliation is manual and error-prone
- No audit trail for disputes
- Password recovery is insecure (plaintext shown back to user — a critical existing flaw)

**Your backend must solve all of these.**

---

## 2. Critical Problems in the Existing Site (Fix These)

| # | Problem | Severity |
|---|---------|----------|
| 1 | Password recovery returns the **plaintext password** — passwords stored unhashed | 🔴 Critical |
| 2 | FPS ID as login = sequential, enumerable IDs — no brute-force protection | 🔴 Critical |
| 3 | No rate limiting on login or signup endpoints | 🔴 Critical |
| 4 | No HTTPS enforcement seen on API calls | 🔴 Critical |
| 5 | Session tokens appear to be long-lived without rotation | 🟠 High |
| 6 | Zero input validation on FPS ID lookup (SQLi surface) | 🔴 Critical |
| 7 | No 2FA for dealer accounts managing PDS data | 🟠 High |
| 8 | "Forgot password" reveals account existence via error message | 🟡 Medium |
| 9 | Browser extension dependency for core workflows is fragile | 🟡 Medium |
| 10 | No offline fallback — dealers in rural Gujarat lose all access on poor network | 🟠 High |

## 3. Tech Stack Decision

### Why This Stack (Not a Default Choice)

| Layer | Choice | Why |
|-------|--------|-----|
| **Runtime** | Node.js 22 (LTS) | Non-blocking I/O; 1M users = high concurrency |
| **Framework** | **Fastify 4** (NOT Express) | 3x faster than Express; schema validation built-in; better for 1M scale |
| **Language** | TypeScript 5 | Catches entire category of bugs before prod; essential at this scale |
| **Primary DB** | PostgreSQL 16 (Supabase or self-hosted) | ACID compliance critical for ration transaction records |
| **Cache** | Redis 7 (Upstash serverless or self-hosted) | Session store, rate limiting, dealer online status |
| **Search** | PostgreSQL FTS (full-text search) | Dealer/beneficiary lookup; no need for Elasticsearch at this stage |
| **Queue** | BullMQ (Redis-backed) | SMS OTP, audit log writes, bulk notification jobs |
| **Auth** | Custom JWT + Refresh Token Rotation | No NextAuth needed; backend-only system |
| **OTP/SMS** | MSG91 or Twilio India | OTP for password reset and 2FA |
| **File Storage** | Cloudflare R2 | Dealer documents, transaction receipts |
| **Deployment** | Docker + Railway/Render (start) → Kubernetes on AWS India (scale) | |
| **API Docs** | Swagger/OpenAPI auto-generated by Fastify | Your friend's frontend needs this |
| **Monitoring** | Pino (logs) + Prometheus + Grafana | |

## 5. Database Schema (PostgreSQL)

```sql
-- ============================================================
-- DEALERS (FPS Shop Owners)
-- ============================================================
CREATE TABLE dealers (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  fps_id        VARCHAR(20) UNIQUE NOT NULL,           -- Primary login ID
  area_id       VARCHAR(20),                           -- Charge/temp shop ID
  full_name     VARCHAR(255) NOT NULL,
  mobile        VARCHAR(15) UNIQUE NOT NULL,
  address       TEXT,
  district      VARCHAR(100),
  taluka        VARCHAR(100),
  village       VARCHAR(100),
  password_hash VARCHAR(255) NOT NULL,                 -- Argon2id hash, NEVER plaintext
  role          VARCHAR(20) DEFAULT 'dealer',          -- dealer | area_officer | admin
  is_active     BOOLEAN DEFAULT TRUE,
  is_verified   BOOLEAN DEFAULT FALSE,
  last_login_at TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_dealers_fps_id ON dealers(fps_id);
CREATE INDEX idx_dealers_mobile ON dealers(mobile);
CREATE INDEX idx_dealers_district ON dealers(district);

-- ============================================================
-- SESSIONS (Refresh Token Store in Redis, metadata here)
-- ============================================================
CREATE TABLE sessions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dealer_id     UUID REFERENCES dealers(id) ON DELETE CASCADE,
  refresh_token_hash VARCHAR(255) NOT NULL,
  user_agent    TEXT,
  ip_address    INET,
  expires_at    TIMESTAMPTZ NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_sessions_dealer_id ON sessions(dealer_id);

-- ============================================================
-- OTP VERIFICATION
-- ============================================================
CREATE TABLE otp_requests (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  mobile      VARCHAR(15) NOT NULL,
  fps_id      VARCHAR(20),
  otp_hash    VARCHAR(255) NOT NULL,           -- Hash the OTP, don't store raw
  purpose     VARCHAR(30) NOT NULL,            -- password_reset | verify_mobile | 2fa
  attempts    SMALLINT DEFAULT 0,
  expires_at  TIMESTAMPTZ NOT NULL,
  used_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_otp_mobile ON otp_requests(mobile);

-- ============================================================
-- BENEFICIARIES (Ration Card Holders under each FPS)
-- ============================================================
CREATE TABLE beneficiaries (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dealer_id       UUID REFERENCES dealers(id),
  ration_card_no  VARCHAR(30) UNIQUE NOT NULL,
  head_of_family  VARCHAR(255) NOT NULL,
  mobile          VARCHAR(15),
  member_count    SMALLINT DEFAULT 1,
  category        VARCHAR(20),                 -- APL | BPL | AAY | PHH
  is_active       BOOLEAN DEFAULT TRUE,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_beneficiaries_dealer_id ON beneficiaries(dealer_id);
CREATE INDEX idx_beneficiaries_ration_card ON beneficiaries(ration_card_no);
-- Full text search on beneficiary name
CREATE INDEX idx_beneficiaries_fts ON beneficiaries USING GIN(to_tsvector('simple', head_of_family));

-- ============================================================
-- STOCK (Commodity Allocation per FPS per Month)
-- ============================================================
CREATE TABLE stock_allocations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dealer_id     UUID REFERENCES dealers(id),
  month         DATE NOT NULL,                 -- First day of month
  commodity     VARCHAR(50) NOT NULL,          -- Rice | Wheat | Sugar | Kerosene | Oil
  allocated_kg  NUMERIC(10,3) NOT NULL,
  lifted_kg     NUMERIC(10,3) DEFAULT 0,
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(dealer_id, month, commodity)
);

CREATE INDEX idx_stock_dealer_month ON stock_allocations(dealer_id, month);

-- ============================================================
-- TRANSACTIONS (Ration Distribution Records)
-- ============================================================
CREATE TABLE transactions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dealer_id       UUID REFERENCES dealers(id),
  beneficiary_id  UUID REFERENCES beneficiaries(id),
  transaction_date DATE NOT NULL,
  month           DATE NOT NULL,
  commodity       VARCHAR(50) NOT NULL,
  quantity_kg     NUMERIC(8,3) NOT NULL,
  price_per_kg    NUMERIC(6,2),
  total_amount    NUMERIC(10,2),
  mode            VARCHAR(20) DEFAULT 'pos',   -- pos | manual | otg
  biometric_auth  BOOLEAN DEFAULT FALSE,
  remarks         TEXT,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_transactions_dealer_date ON transactions(dealer_id, transaction_date);
CREATE INDEX idx_transactions_beneficiary ON transactions(beneficiary_id);
CREATE INDEX idx_transactions_month ON transactions(month);

-- ============================================================
-- AUDIT LOG (Immutable — append only)
-- ============================================================
CREATE TABLE audit_logs (
  id          BIGSERIAL PRIMARY KEY,
  dealer_id   UUID,
  action      VARCHAR(100) NOT NULL,
  entity      VARCHAR(50),
  entity_id   UUID,
  old_data    JSONB,
  new_data    JSONB,
  ip_address  INET,
  user_agent  TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_audit_dealer ON audit_logs(dealer_id);
CREATE INDEX idx_audit_created ON audit_logs(created_at);

-- ============================================================
-- NOTIFICATIONS
-- ============================================================
CREATE TABLE notifications (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dealer_id   UUID REFERENCES dealers(id),
  title       VARCHAR(255) NOT NULL,
  body        TEXT NOT NULL,
  type        VARCHAR(30),                     -- info | warning | alert
  is_read     BOOLEAN DEFAULT FALSE,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_notifications_dealer ON notifications(dealer_id, is_read);
```

## 7. Authentication & Security Design

### Token Strategy
```
Login → Access Token (15 min TTL, JWT, stateless)
      + Refresh Token (30 day TTL, stored in Redis + HttpOnly cookie)

Refresh → New Access Token + Rotated Refresh Token (old one invalidated)

Logout → Delete Redis key for refresh token
```

### Password Security
- Hash with **Argon2id** (not bcrypt — better against GPU attacks)
- Parameters: `memory=65536`, `iterations=3`, `parallelism=4`
- Never return, log, or store plaintext passwords anywhere
- Password reset via OTP only (no "show me my password")

### Rate Limiting (per Redis)
```
POST /auth/login         → 5 attempts / 15 min / IP
POST /auth/login         → 10 attempts / 1 hour / FPS ID
POST /auth/forgot-*      → 3 requests / 30 min / mobile
POST /dealers/register   → 3 requests / 1 hour / IP
All other endpoints      → 100 req / min / user
```

### OTP Design
- 6-digit numeric OTP
- TTL: 10 minutes
- Max attempts: 3 (then OTP invalidated, must request new)
- Store only the **hash** of the OTP (SHA-256), never the raw value
- Rate limit: 3 OTPs / 30 min per mobile

### Input Validation
- All request bodies validated via Zod schemas at route level
- FPS ID format: must match `/^\d{5,20}$/`
- Mobile: must be valid Indian 10-digit number
- All string inputs sanitized for SQL injection at query level (parameterized queries)

## 8. Scalability Architecture (1M Users)

### Traffic Estimation
```
1,000,000 dealers × avg 5 req/day = 5,000,000 req/day
Peak factor (morning 8-11am): 10x = ~580 req/sec peak

Fastify handles ~30,000 req/sec on a single core.
You need ~1 instance for 10x headroom at peak.
But for HA + reliability, run minimum 3 instances.
```

### Deployment Tiers

**Tier 1: Launch (0–50k users)**
```
Railway / Render:
  - 2x Node.js instances (auto-scaled)
  - 1x PostgreSQL (Neon or Supabase — managed)
  - 1x Redis (Upstash — serverless)
Cost: ~$50–100/month
```

**Tier 2: Growth (50k–300k users)**
```
AWS India (ap-south-1, Mumbai):
  - ECS Fargate: 3–6 tasks (auto-scaled via CPU/RPS)
  - RDS PostgreSQL: db.t4g.large with 1 read replica
  - ElastiCache Redis: cache.r7g.large
  - CloudFront CDN: for static assets, API caching
Cost: ~$400–800/month
```

**Tier 3: Scale (300k–1M users)**
```
AWS India:
  - EKS (Kubernetes): 6–20 pods, HPA on CPU+RPS
  - RDS PostgreSQL: db.r7g.xlarge + 2 read replicas
  - ElastiCache Redis cluster mode
  - Database connection pooling via PgBouncer
  - Separate read/write DB connections in app
Cost: ~$1,500–3,000/month
```

### Caching Strategy
```
Redis Cache Layers:
  1. Dealer session store (access token metadata): TTL 15 min
  2. Dealer profile: TTL 5 min (invalidate on update)
  3. Dashboard stats (total dealers, online count): TTL 30 sec
  4. FPS ID existence check (for signup): TTL 1 hour
  5. Rate limit counters: sliding window TTL
```

### Database Query Optimization
- All foreign keys indexed
- Pagination uses cursor-based (not OFFSET — OFFSET kills performance at scale)
- Transactions table partitioned by `month` (PostgreSQL declarative partitioning)
- Audit log table uses `BIGSERIAL`, never deleted — archive old partitions to S3

## 9. Real-Time Online/Offline Tracking

The existing site shows "Online" and "Offline" counts. Implement this properly:

```
On login:
  → Redis: SET dealer:{id}:online 1 EX 900  (15 min TTL)

On any authenticated request:
  → Redis: EXPIRE dealer:{id}:online 900    (sliding window)

On logout:
  → Redis: DEL dealer:{id}:online

Dashboard stats endpoint:
  → Redis: KEYS dealer:*:online             (avoid in prod — use SET instead)
  → Better: Redis SET "online_dealers" tracking with SADD/SREM

Production approach:
  → Redis SADD online_dealers {dealer_id}   on login/heartbeat
  → Redis SREM online_dealers {dealer_id}   on logout
  → Cron job every 5 min: clean up expired sessions from set
  → GET /dealers/:id/stats returns SCARD online_dealers
```

## 13. RBAC (Role-Based Access Control)

```
Roles:
  dealer       → Can only access their own data
  area_officer → Can view dealers in their area (read-only)
  admin        → Full access, including bulk operations

Middleware:
  authenticate.ts  → Verifies JWT, attaches req.user
  authorize.ts     → Takes required role, throws 403 if insufficient

Usage:
  fastify.get('/admin/dealers', {
    preHandler: [authenticate, authorize('admin')]
  }, handler)
```

## 18. What NOT to Do

- ❌ Don't use Express — Fastify is measurably better at this scale
- ❌ Don't store passwords in plaintext or with MD5/SHA1
- ❌ Don't use OFFSET pagination — it breaks at 100k+ rows
- ❌ Don't put JWT secrets in code — env only, rotated quarterly
- ❌ Don't skip input validation — PDS data is sensitive government data
- ❌ Don't use `SELECT *` in production queries — always name columns
- ❌ Don't let refresh tokens live in localStorage (use HttpOnly cookies)
- ❌ Don't build admin features before core dealer auth is solid
